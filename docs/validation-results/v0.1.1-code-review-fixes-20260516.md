# v0.1.1 Code Review Fix Notes

**Date:** 2026-05-16
**Scope:** Local source review and unit/integration validation only.
**Cloud validation:** Not run. No AWS S3, BOS, OSS, or other real provider commands were executed.

## Summary

Two issues found during third-party v0.1.0 source review were confirmed and fixed for v0.1.1:

1. `src/tasks_s3.rs` used a hard-coded 5 second per-page watchdog around `ListObjectsV2` paginator `stream.next()`.
2. `profile = "bos"` still selected path-style addressing by default, while product documentation now positions BOS as virtual-hosted-first.

## Fixes

### Per-page timeout

The per-page paginator watchdog now uses `S3Config.operation_timeout_secs`, matching the SDK operation, read, and attempt timeout configuration.

Stream timeouts are treated as transient errors again, but retries are bounded by `S3Config.max_attempts` at the task orchestration layer. This preserves retry behavior without reintroducing the previous infinite retry risk.

### BOS addressing preset

The built-in `bos` profile now defaults to virtual-hosted addressing and the default BOS S3-compatible endpoint.

Path-style remains available for legacy or diagnostic use through explicit configuration:

```bash
--addressing-style path
```

or:

```bash
--force-path-style
```

`--endpoint-url` no longer implicitly forces path-style addressing. This avoids surprising behavior when users specify an endpoint and also request virtual-hosted addressing.

## BOS Compatibility Position

The BOS ListObjectsV2 `start_after` + `continuation-token` behavior remains a BOS service-side compatibility issue. v0.1.1 does not add a BOS-specific pagination workaround and does not alter the standard high-performance S3-compatible listing path.
